# Формат отчёта о найденных багах API

Расхождения API со спекой и подозрительное поведение, найденные при прогоне,
докладывайте пользователю в таком виде (по убыванию severity).

## Severity

- **Critical** — потеря денег/данных, обход авторизации, доступ к чужим ресурсам.
- **High** — функция не работает; спека врёт о ключевом поведении.
- **Medium** — принимаются невалидные данные; неверные коды ошибок; недокументированные обязательные поля.
- **Low** — расхождения форматов (типы, регистр имён полей), косметика.

## Шаблон записи

> ### [Medium] PUT /users/me принимает невалидный email
> **Ожидание (спека):** 400 «Ошибка валидации» на email не по формату.
> **Факт:** 200, профиль обновлён на `not-an-email`; последующие ответы отдают его же.
> **Воспроизведение:** `PUT /users/me` с телом `{"email": "not-an-email"}` под валидным токеном.
> **Покрытие:** тест `test_update_me_invalid_email`, `xfail(strict=True)` — просигналит, когда починят.

## Пример итоговой сводки (реальный прогон RV Booker)

| Severity | Находка | Покрытие в тестах |
|---|---|---|
| Medium | PUT /users/me принимает невалидный email (200 вместо 400) | strict xfail |
| Medium | Оплата проходит с любой картой — валидации нет (стаб?) | happy path покрыт; негатив недостижим |
| Low | Числовые поля БД приходят строками (`base_price: "5500.00"`) | Decimal в моделях + комментарий |
| Low | Даты брони — datetime вместо documented `format: date` | datetime в модели + комментарий |
| Low | GET /users/me отдаёт `createdAt` (camelCase), без `is_active` | отдельная модель UserProfile |

Каждая находка — либо тест (xfail для подтверждённых багов), либо комментарий
в модели, либо явное «недостижимо» с причиной. Находок без следа в коде быть не должно.
