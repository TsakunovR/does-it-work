# Как извлекать тест-кейсы из разных источников

## OpenAPI / Swagger

1. Получите спеку: файл в репозитории (`openapi.yaml|json`) или `GET {base_url}/openapi.json`.
2. Для каждого эндпоинта извлеките: метод, путь, параметры, схему тела, схемы ответов, коды ошибок, security-схему.
3. Из `components.schemas` сгенерируйте Pydantic-модели ответов (`extra="forbid"`).
   Ограничения (`minLength`, `minimum`, `maximum`, `required`, `format`) — это готовый
   список негативных кейсов для `parametrize`: каждое ограничение → значение сразу за границей.
4. Из перечня кодов ответов: каждый документированный 4xx — отдельный негативный тест;
   недокументированные ответы, которые вы обнаружите при запуске, — повод для баг-репорта.
5. Порядок покрытия: сначала happy path каждого эндпоинта, потом связки (создал → прочитал →
   удалил → 404), потом негативные.
6. Прежде чем писать модели по спеке, **допросите живой стенд** (curl по основным
   эндпоинтам): реальные ответы часто расходятся со спекой (числа строками из БД,
   camelCase вместо snake_case, другой формат ошибок). Модель фиксирует фактический
   контракт, каждое расхождение — комментарий в модели + пункт в отчёте пользователю.
7. Кодогенерация моделей (`datamodel-code-generator`) оправдана при десятках схем —
   один прогон, потом причесать руками. Для типичного API модели лучше писать вручную:
   генератор тянет всю спеку целиком (шум), а главное — слепо повторяет её ошибки,
   тогда как рукописная модель отражает проверенное на стенде поведение (см. п. 6).

## Исходный код сервиса

1. Найдите роуты: FastAPI (`@app.get/post`, `APIRouter`), Flask (`@app.route`), Django (`urls.py`).
2. Прочитайте модели запросов/ответов и валидаторы — из них следуют границы для негативных кейсов.
3. Прочитайте бизнес-логику обработчиков: ветки `if`/`raise HTTPException` — каждая ветка
   ошибки должна быть покрыта тестом.
4. Найдите зависимости (auth, БД, внешние вызовы) — они определяют, что нужно в фикстурах
   и что мокать через respx.
5. Если приложение — FastAPI/Flask и доступно как модуль, включите интеграционный режим
   in-process (см. conftest шаблона): тесты не требуют развёрнутого стенда.

## Текстовое описание / требования

1. Каждое требование переформулируйте как проверяемое утверждение
   «при <запросе> API возвращает <код> и <свойство ответа>».
2. Неоднозначности не додумывайте — задайте вопросы пользователю до генерации.
3. К каждому позитивному требованию сгенерируйте негативные: а что, если поле пустое,
   не тот тип, нет прав, ресурс не существует.

## curl / Postman / HAR

1. curl: разберите метод, URL, заголовки (`-H`), тело (`-d/--data`).
2. Postman-коллекция: `item[].request` — метод, `url.raw`, `header`, `body.raw`;
   переменные `{{var}}` перенесите в `config.py` / фикстуры.
3. HAR: `log.entries[].request/response` — реальные пары запрос-ответ; фактический ответ
   используйте как основу Pydantic-модели, но сверьте с документацией (в HAR может быть
   устаревшее или частное поведение).
4. Захардкоженные токены и id из примеров НЕ переносите в тесты — токены в `Settings`,
   id — через фикстуры создания.
