# Режимы «Стабилизация» и «Ревью» существующих тестов

Оба режима работают с чужим кодом: сначала факты (запуск), потом выводы.
Никаких правок без диагноза, никаких вердиктов без прогона.

## Режим «Стабилизация» (флакающие тесты)

### 1. Воспроизведи

Один прогон ничего не доказывает. Запусти подозрительный набор 5–10 раз подряд
(`for i in $(seq 5); do pytest ...; done`), фиксируй частоту падений и конкретные
тесты. Если флак не воспроизвёлся — расширь до полного сьюта: причина может быть
в соседях (изоляция), а не в самом тесте.

### 2. Классифицируй причину

| Симптом | Вероятная причина | Рецепт |
|---|---|---|
| Таймауты/сетевые ошибки на разных тестах | Нестабильный стенд или сеть | `retry` на транспортном уровне (utils/retry, backoff); таймауты в конфиг, не в код |
| Element not found / stale element / пустая страница | Ожидания: `sleep`, `networkidle`, обращение до готовности | `wait_until` / `expect(...).to_be_visible()` / `shouldBe(visible)`; локаторы на стабильные id (см. Gotchas web-веток) |
| Поодиночке зелёный, в полном прогоне падает | Изоляция: общие данные, зависимость от порядка, session-фикстуры с состоянием | Уникальные данные фабриками, очистка в teardown, тест не должен читать чужие результаты |
| 400/409 «занято/существует/недоступно» | Общий стенд: конфликт с чужими данными | Ретрай с **новыми случайными данными**, повторять только по ожидаемой ошибке (паттерн `create_booking` в live-api-patterns) |
| Локально зелёный, в CI красный (или наоборот) | Окружение: headless, вьюпорт, версии браузера, медленный раннер | Зафиксируй окружение в конфиге; артефакты при падении (скриншот/trace) покажут разницу |
| Изредка другой ответ API при тех же входных | Race condition или баг сервиса | Это находка, а не флак теста: воспроизведи руками, покажи пользователю (возможен баг-репорт) |

### 3. Чини причину, не симптом

Поднять таймаут, добавить `sleep`, перезапустить — лечение симптома; флак вернётся.
Исключение: таймаут, заниженный относительно реального SLA операции, — тогда это
и есть причина.

### 4. Докажи стабильность

После починки — минимум **5 зелёных прогонов подряд** (при гонках и изоляции — 10+).
Пока не доказано повторами, задача не закрыта.

### 5. Тупик → карантин, но с долгом

Если причина не находится за разумное время: маркер/тег `flaky` + комментарий
с симптомом и гипотезой + договорённость с пользователем, когда вернуться.
Карантин без комментария и срока — это выключенный тест, а не карантин.

Red flags режима: «перезапущу — пройдёт», «подниму таймаут и всё», «отключу этот тест»,
«починил, один раз прошло — готово».

## Режим «Ревью» (оценка качества чужих тестов)

### 1. Сначала запусти

Ревью без запуска — не ревью, а чтение. Прогони сьют (при подозрениях на флаки —
2–3 раза), зафиксируй факты: сколько тестов, сколько падает, длительность,
предупреждения. Красный сьют меняет приоритеты ревью: сначала «почему красный».

### 2. Пройди чек-листы

- Чек-лист «Самопроверка качества» из SKILL.md (шаг 5) — по каждому тестовому файлу.
- Дополнительно для чужого кода:
  - HTTP-вызовы прямо в тестах вместо клиентского слоя?
  - Дублирование проверок/сетапа вместо утилит и фикстур?
  - Конфиг и URL захардкожены вместо env-переменных?
  - **Секреты (токены, пароли) в коде?**
  - Есть ли негативные кейсы и проверка контрактов, или только happy path?
  - UI: селекторы хрупкие (xpath по индексам)? состояние готовится через UI вместо API?

### 3. Классифицируй находки по severity

- **Critical** — секреты в коде; тесты, которые не могут упасть (нет ассертов,
  ассерт truthiness); зависимость от порядка выполнения.
- **High** — нет изоляции данных; `sleep`; хрупкие селекторы; teardown отсутствует
  (мусор копится на стенде).
- **Medium** — слабые ассерты (только статус без тела); копипаста вместо слоёв;
  нет негативных кейсов; нет контрактов.
- **Low** — стиль, именование, отсутствие маркеров/severity, структура файлов.

### 4. Отчёт — сначала, правки — потом

Формат: таблица находок (severity → файл → проблема → рецепт из этого скилла) +
отдельно «что уже хорошо» + план внедрения по приоритету. **Не переписывай чужие
тесты без явного запроса** — сначала отчёт и согласование объёма с пользователем;
дальше правки идут пакетами от Critical к Low, с зелёным прогоном после каждого пакета.
